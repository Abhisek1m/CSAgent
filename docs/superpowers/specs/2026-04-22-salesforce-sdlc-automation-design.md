# CSAgent вЂ” Salesforce AI SDLC Automation Design Spec
Date: 2026-04-22

## Overview

CSAgent is a Claude Code project that uses Claude AI to automate the first three phases of the Salesforce SDLC вЂ” Requirements, Design, and Development. It lives at `e:/AIapps/CSAgent/` and is opened as a standalone VS Code workspace. Claude generates all artifacts (user stories, solution designs, Apex code, LWC components, Flow descriptions) via slash commands backed by specialist agents. All AI-generated Salesforce components are prefixed with `CS` to distinguish them from human-written code.

**MVP Scope:** Requirements в†’ Design в†’ Development (3 phases)
**Not in MVP:** Testing automation, CI/CD, deployment pipelines

---

## The Golden Rule

All agents enforce this priority order without exception:

1. **Standard out-of-the-box** вЂ” use what Salesforce already provides natively
2. **Declarative configuration** вЂ” Flows, Validation Rules, Formula Fields, Page Layouts
3. **Custom code** вЂ” Apex, LWC, only when 1 and 2 cannot meet the requirement

Every design decision that reaches code must include an explicit **Why Code?** justification.

## Naming Conventions (CS Prefix)

All components generated by CSAgent are prefixed with `CS` to be instantly distinguishable from human-written code. API names are immutable in Salesforce once deployed вЂ” this convention is set from day one.

| Component | Convention | Example |
|---|---|---|
| Custom Object | `CS_` + PascalCase + `__c` | `CS_ServiceRequest__c` |
| Custom Field | `CS_` + PascalCase + `__c` | `CS_Priority__c` |
| Apex Class | `CS` + PascalCase | `CSAccountService` |
| Apex Trigger | `CS` + PascalCase + `Trigger` | `CSAccountTrigger` |
| LWC Component | `cs` + PascalCase | `csAccountHierarchy` |
| Flow | `CS_` + PascalCase | `CS_AccountValidation` |
| Test Class | `CS` + ClassName + `Test` | `CSAccountServiceTest` |

## Specialist Agents

### Story Agent (`story-agent.md`)

**Role:** Requirements specialist
**Input:** Plain English business requirement
**Output:** Structured user stories saved to `stories/YYYY-MM-DD-<name>.md`

Behavior:
- Produces user stories in standard format: `As a [Salesforce role], I want [feature] so that [benefit]`
- Includes acceptance criteria, priority (High/Medium/Low), and story point estimate
- Maps requirements to Salesforce roles: Admin, Sales Rep, Service Agent, Developer, Executive
- Flags requirements that map to standard Salesforce features (e.g., "track emails" в†’ "standard Activity object exists, no custom solution needed")
- Groups related stories into epics when a requirement has multiple sub-features
- Uses Claude's built-in Salesforce knowledge вЂ” no org queries needed at this phase

### Design Agent (`design-agent.md`)

**Role:** Architecture + data model specialist
**Input:** Story file path (`stories/YYYY-MM-DD-<name>.md`)
**Output:** Solution design saved to `designs/YYYY-MM-DD-<name>.md`

Behavior:
- Loads `cache/org-context.json` first to check existing custom objects/fields before proposing new ones
- Applies the Golden Rule explicitly: evaluates standard objects в†’ config в†’ code in that order
- Produces:
  - Object model (standard objects to reuse, CS-prefixed custom objects to create)
  - Field definitions (API name, type, required, default)
  - Relationship diagram (text-based)
  - Automation recommendation: Flow vs. Validation Rule vs. Apex trigger (with justification)
  - Component breakdown: standard page vs. LWC vs. Flow screen
- Every code recommendation includes a **Why Code?** section explaining why declarative options were ruled out
- Uses Claude's built-in Salesforce knowledge for standard object schemas

### Dev Agent (`dev-agent.md`)

**Role:** Apex / LWC / Flow code generation specialist
**Input:** Design file path + artifact type + component name
**Output:** Production-ready code files in `force-app/main/default/`

Behavior:
- Loads `cache/org-context.json` to avoid naming conflicts with existing classes/components
- Enforces CS prefix on every generated artifact вЂ” refuses to generate without it
- Generates Apex following best practices:
  - Bulkified triggers (no SOQL/DML in loops)
  - Proper sharing model (`with sharing` by default)
  - Separation of concerns (trigger в†’ handler class pattern)
  - Every Apex class gets a companion `CSClassNameTest` with minimum 75% coverage target
- Generates LWC with:
  - Wire adapters for data access (no imperative Apex when wire works)
  - Proper error handling and loading states
  - Accessible markup (ARIA labels)
- Generates Flow descriptions + metadata XML structure
- Every generated file includes a header comment block:
  ```
  // Generated by CSAgent | [date]
  // Standard/Config option considered: [reason ruled out]
  ```
- Can run `sf sobject describe` via CLI to verify field existence before generating

## Org Context Cache

### Purpose
Avoid re-querying the org for metadata already known. Claude's prompt caching is triggered by injecting the static cache content at the top of `CLAUDE.md` вЂ” repeated calls with the same cache get ~90% token cost reduction.

### Cache Structure (`cache/org-context.json`)
```json
{
  "orgAlias": "",
  "apiVersion": "62.0",
  "lastRefreshed": "",
  "customObjects": [],
  "customFields": {},
  "apexClasses": [],
  "apexTriggers": [],
  "lwcComponents": [],
  "installedPackages": []
}
```

### Cache Rules
- Agents read from cache first вЂ” never query the org directly unless cache is missing
- `/sf-org-inspect` runs `sf sobject list`, `sf apex list`, and related CLI commands once and writes results
- `/sf-refresh-cache` re-runs inspection when the org changes (new objects deployed, packages installed)
- Claude's built-in knowledge covers all standard Salesforce objects вЂ” cache stores only org customizations

## Token Optimization

Borrowed from the everything-claude-code patterns:

- `MAX_THINKING_TOKENS: "10000"` вЂ” caps extended thinking to prevent runaway costs
- Org context cache injected as static block in `CLAUDE.md` вЂ” triggers Claude prompt cache hits
- SessionStart hook silently loads org context into conversation without user prompting
- Agents use Claude's built-in Salesforce knowledge wherever possible, minimizing tool calls
