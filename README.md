# CSAgent — Salesforce AI SDLC Automation

CSAgent is a [Claude Code](https://claude.ai/code) project that automates the Salesforce development lifecycle using AI agents. Give it a plain English business requirement and it produces production-ready Salesforce metadata — custom objects, fields, flows, permission sets, and deployment instructions — ready to deploy to your org.

---

## How It Works

```
Plain English Requirement
        │
        ▼
  /sf-story          → stories/             (structured user stories)
        │
        ▼
  /sf-design         → designs/             (solution design + data model)
        │
   ┌────┼────────────────────┐
   ▼    ▼                    ▼
/sf-object  /sf-flow    /sf-apex or /sf-lwc
   │    │                    │
   ▼    ▼                    ▼
force-app/main/default/  (deployable metadata)
        │
        ▼
/sf-permission-set   → permissionsets/
        │
        ▼
/sf-admin-setup      → designs/setup-instructions/  (manual Setup UI steps)
        │
        ▼
  sf project deploy start  →  Salesforce Org
```

## Setup

### 1. Clone and open

```bash
git clone <this-repo>
cd CSAgent
code .   # or open in Cursor / any editor
```

### 2. Connect your Salesforce org

Run this in your terminal (not inside Claude Code):

```bash
sf org login web --alias csagent-dev
```

### 3. Start Claude Code and populate the org cache

```bash
claude
```

Then in Claude Code:

```
/sf-org-inspect
```

This populates `cache/org-context.json` with your org's custom objects, Apex classes, triggers, and installed packages. CSAgent uses this to avoid naming conflicts when generating new components.

## The Full Workflow

### Step 1 — Write user stories

```
/sf-story "Track customer satisfaction scores after support cases are closed. 
Each score should be linked to a Case, have a 1–5 rating, a date, and customer notes."
```

Saves to: `stories/YYYY-MM-DD-<feature>.md`

---

### Step 2 — Generate a solution design

```
/sf-design stories/2026-04-22-customer-satisfaction-score.md
```

The design-agent applies the **Golden Rule** (Standard → Config → Code) and produces:
- Data model with full field definitions
- Automation strategy (Flow preferred over Apex)
- Relationship diagram
- Implementation order

Saves to: `designs/YYYY-MM-DD-<feature>.md`

---

### Step 3 — Generate metadata

Run in this order — each component depends on the previous:

```
/sf-object designs/2026-04-22-customer-satisfaction-score.md CS_SatisfactionScore__c
```
```
/sf-permission-set designs/2026-04-22-customer-satisfaction-score.md CS_ServiceAgentPermissions
```
```
/sf-flow designs/2026-04-22-customer-satisfaction-score.md CS_CreateSatisfactionScoreOnCaseClosure
```
```
/sf-admin-setup designs/2026-04-22-customer-satisfaction-score.md
```

---

### Step 4 — Deploy to Salesforce

```bash
# 1. Object and fields first (permission set and flow depend on this)
sf project deploy start --source-dir force-app/main/default/objects/CS_SatisfactionScore__c --target-org csagent-dev

# 2. Permission set
sf project deploy start --source-dir force-app/main/default/permissionsets --target-org csagent-dev

# 3. Flow
sf project deploy start --source-dir force-app/main/default/flows --target-org csagent-dev
```

---

### Step 5 — Complete manual Setup steps

Open the file generated by `/sf-admin-setup`:

```
designs/setup-instructions/YYYY-MM-DD-<feature>-setup.md
```

This covers activities that truly **cannot** be deployed via metadata:
- Escalation Rules (not a Salesforce metadata type)
- Approval Processes (complex, keep manual)
- Reports / Dashboards (not deployable via `sf project deploy`)
- OWD / Sharing Settings (keep manual — triggers org-wide sharing recalculation)
- Permission set assignment to users

> **Note:** Page Layouts, Custom Report Types, and Email Templates ARE deployable. `/sf-admin-setup` generates their metadata XML automatically — no Setup UI needed for those.

## Command Reference

| Command | Input | What it generates |
|---------|-------|-------------------|
| `/sf-org-inspect` | *(none)* | `cache/org-context.json` — org metadata snapshot |
| `/sf-refresh-cache` | *(none)* | Overwrites `cache/org-context.json` with fresh data |
| `/sf-story` | `"requirement text"` | `stories/YYYY-MM-DD-<name>.md` |
| `/sf-design` | `stories/<file>.md` | `designs/YYYY-MM-DD-<name>.md` |
| `/sf-object` | `designs/<file>.md CS_ObjectName__c` | Object + fields + validation rules metadata |
| `/sf-permission-set` | `designs/<file>.md CS_PermSetName` | Permission set (+ group + sharing rules if needed) |
| `/sf-flow` | `designs/<file>.md CS_FlowName` | Record-triggered or screen Flow metadata |
| `/sf-apex` | `designs/<file>.md CSClassName` | Apex class + test class |
| `/sf-lwc` | `designs/<file>.md csComponentName` | Lightning Web Component |
| `/sf-admin-setup` | `designs/<file>.md` | Page layout + report type + email template metadata; Setup UI instructions for non-deployable activities |

## Naming Conventions

All components generated by CSAgent use the `CS_` prefix. This distinguishes AI-generated code from human-written code and prevents naming conflicts.

| Component | Convention | Example |
|-----------|-----------|---------|
| Custom Object | `CS_` + PascalCase + `__c` | `CS_SatisfactionScore__c` |
| Custom Field | `CS_` + PascalCase + `__c` | `CS_Rating__c` |
| Apex Class | `CS` + PascalCase | `CSAccountService` |
| Apex Trigger | `CS` + PascalCase + `Trigger` | `CSCaseTrigger` |
| LWC Component | `cs` + PascalCase | `csSatisfactionForm` |
| Flow | `CS_` + PascalCase | `CS_CreateSatisfactionScore` |
| Test Class | `CS` + ClassName + `Test` | `CSAccountServiceTest` |
| Permission Set | `CS_` + PascalCase | `CS_ServiceAgentPermissions` |

## The Three Agents

### story-agent
Converts plain English requirements into structured Salesforce user stories. Flags when a standard Salesforce feature already covers a requirement before recommending custom development.

### design-agent
Converts user stories into a complete solution design. Applies the Golden Rule — always evaluates standard features first, then declarative configuration, then custom code as a last resort. Reads the org context cache to avoid conflicts.

### dev-agent
Generates production-ready Salesforce DX metadata from a solution design. Knows all 18 Salesforce field types, standard Flow patterns, permission set rules (e.g. required fields cannot have FLS), and admin setup instructions for 12 activity types.

---

## The Golden Rule

Before generating any component, CSAgent evaluates options in this order:

1. **Standard out-of-the-box** — does Salesforce already do this natively?
2. **Declarative configuration** — Flows, Validation Rules, Formula Fields, Permission Sets?
3. **Custom code** — Apex or LWC — only if 1 and 2 genuinely cannot meet the requirement

When Apex or LWC is recommended, the design always includes a **Why Code?** section explaining why declarative options were ruled out.
